# Change-set for #22 — Add rate limiting per authenticated user (in-memory token bucket)

## Scope

Add a pure-Python in-memory token bucket rate limiter with two tiers: 60 req/min per authenticated user (all authed endpoints), and 10 req/min per client IP for `/auth/register` and `/auth/login`. No external dependencies. `GET /health` is explicitly excluded from rate limiting.

## Files to touch

- Create: `src/todo_api/rate_limit.py` — new module containing `TokenBucket` class, `_buckets` dict, `_get_bucket` helper, `authed_rate_limit` dependency, `anon_rate_limit` dependency, and `reset_buckets` testing helper
- Modify: `src/todo_api/app.py` — replace `Depends(get_current_user)` with `Depends(authed_rate_limit)` on all authed endpoints; add `_: str = Depends(anon_rate_limit)` to `POST /auth/register` and `POST /auth/login`; import `authed_rate_limit` and `anon_rate_limit` from `rate_limit`
- Modify: `tests/conftest.py` — rename fixture to `isolated_state`, add `rate_limit.reset_buckets()` call, import `rate_limit` module
- Create: `tests/test_rate_limit_unit.py` — 6 unit tests for `TokenBucket` directly
- Create: `tests/test_rate_limit_authed.py` — 5 integration tests for authed rate limiting via FastAPI
- Create: `tests/test_rate_limit_anon.py` — 4 integration tests for IP-keyed rate limiting on auth endpoints

## Public API / contract changes

**Before:** All authenticated endpoints return 200/201/204 on success; `/auth/register` and `/auth/login` return 201/200 on success.

**After:** Every authenticated endpoint may additionally return `429 Too Many Requests` with header `Retry-After: <seconds>` when the per-user token bucket is exhausted (>60 requests/min). `/auth/register` and `/auth/login` may additionally return `429` when the per-IP bucket is exhausted (>10 requests/min).

**New dependency signatures:**
```python
# rate_limit.py
class TokenBucket:
    def __init__(self, capacity: int, refill_rate_per_second: float) -> None: ...
    def consume(self, n: int = 1, now: float | None = None) -> bool: ...

def authed_rate_limit(user_id: int = Depends(get_current_user)) -> int: ...
def anon_rate_limit(request: Request) -> str: ...
def reset_buckets() -> None: ...
```

**`Retry-After` value:** `math.ceil(1 / refill_rate)`, minimum 1 second. For authed (refill=1.0/s) → `1`. For anon (refill=10/60≈0.1667/s) → `6`.

## Test plan

**New tests:**

- `tests/test_rate_limit_unit.py` (6 tests):
  1. `test_fresh_bucket_allows_capacity_requests` — capacity=5, 5 consumes all return True
  2. `test_bucket_denies_when_exhausted` — 6th consume returns False
  3. `test_bucket_refills_over_time` — advance `now` by 1s with refill_rate=1.0, one more consume succeeds
  4. `test_bucket_refill_caps_at_capacity` — consume 1, advance 10s with rate=1.0, capacity=5 → only 5 tokens available
  5. `test_consume_n` — `consume(3)` True then `consume(3)` False (only 2 left)
  6. `test_consume_negative_raises` — `consume(-1)` and `consume(0)` raise `ValueError`

- `tests/test_rate_limit_authed.py` (5 tests):
  1. `test_authed_user_gets_60_requests_per_minute` — 60 `GET /todos` → all 200
  2. `test_authed_user_hits_429_on_61st` — 61st → 429, `Retry-After` header present and integer-valued
  3. `test_two_users_have_independent_buckets` — user A exhausts; user B first request succeeds
  4. `test_bucket_resets_on_refill` — monkeypatch `time.monotonic`, advance past refill, next request succeeds
  5. `test_health_is_not_rate_limited` — 100 `GET /health` → all 200

- `tests/test_rate_limit_anon.py` (4 tests):
  1. `test_register_limited_to_10_per_minute_per_ip` — 10 attempts succeed or 409; 11th → 429
  2. `test_login_shares_bucket_with_register` — 5 register + 6 login from same IP; 11th request → 429
  3. `test_different_ips_have_independent_buckets` — monkeypatch `request.client.host` via dependency override to simulate different IPs
  4. `test_429_response_includes_retry_after` — assert header present and value is integer string ≥ 1

**Existing tests must pass:** All pre-existing tests in `tests/test_*.py`. The `conftest.py` fixture rename from `isolated_db` to `isolated_state` is backward-compatible since it is `autouse=True` and not referenced by name in other tests.

## Risk justification

This change modifies the public HTTP API surface by introducing a new 429 response on every authenticated endpoint and on `/auth/register` and `/auth/login`. Per the risk rubric, any public API surface change classifies as high.

## Open questions

- The issue says `test_login_shares_bucket_with_register` — 5 register + 6 login should hit 429 on the 11th request. Registering the same email 5 times will 409 after the first; the test should use unique emails or accept 409 as a "counted" request. The Implementer should use unique emails per register attempt and count all responses toward the IP bucket total.
- `request.client` may be `None` in test environments using `TestClient`; the Implementer should guard with a fallback key (e.g., `"testclient"`) or use a dependency override for IP in anon tests.
